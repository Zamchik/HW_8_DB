# Сущности проекта «KeyMarket»

## 1. User (Пользователь)

Описание: зарегистрированный пользователь платформы. Может быть покупателем, продавцом или администратором.

### Поля

| Поле          | Тип            | Обязательное | Описание |
|---------------|----------------|--------------|----------|
| id            | SERIAL         | Да           | Уникальный идентификатор (PK) |
| email         | VARCHAR(255)   | Да           | Email, уникальный |
| password_hash | VARCHAR(255)   | Да           | Хеш пароля |
| role          | VARCHAR(20)    | Да           | Роль: buyer, seller, admin (по умолчанию buyer) |
| balance       | NUMERIC(10,2)  | Да           | Баланс в рублях, по умолчанию 0 |
| avatar_url    | TEXT           | Нет          | Ссылка на аватар |
| is_banned     | BOOLEAN        | Да           | Забанен ли пользователь, по умолчанию false |
| created_at    | TIMESTAMP      | Да           | Дата регистрации, по умолчанию now() |

## 2. Category (Категория)

Описание: категория (жанр/тип) игры для группировки товаров.

### Поля

| Поле | Тип           | Обязательное | Описание |
|------|---------------|--------------|----------|
| id   | SERIAL        | Да           | Уникальный идентификатор (PK) |
| name | VARCHAR(100)  | Да           | Название категории, уникальное |
| slug | VARCHAR(100)  | Да           | URL-friendly идентификатор, уникальный |

## 3. Product (Товар)

Описание: карточка игры, которую продавец выставляет на продажу. Количество доступных ключей (stock) — вычисляемое поле.

### Поля

| Поле        | Тип            | Обязательное | Описание |
|-------------|----------------|--------------|----------|
| id          | SERIAL         | Да           | Уникальный идентификатор (PK) |
| seller_id   | INTEGER        | Да           | ID продавца, FK → User.id |
| category_id | INTEGER        | Да           | ID категории, FK → Category.id |
| title       | VARCHAR(200)   | Да           | Название игры |
| description | TEXT           | Нет          | Описание игры |
| price       | NUMERIC(10,2)  | Да           | Цена в рублях |
| rating      | NUMERIC(3,2)   | Да           | Средний рейтинг, по умолчанию 0 |
| status      | VARCHAR(20)    | Да           | Статус: active, inactive, banned (по умолчанию active) |
| created_at  | TIMESTAMP      | Да           | Дата создания, по умолчанию now() |

## 4. ProductKey (Ключ продукта)

Описание: конкретный ключ активации Steam, привязанный к товару.

### Поля

| Поле       | Тип            | Обязательное | Описание |
|------------|----------------|--------------|----------|
| id         | SERIAL         | Да           | Уникальный идентификатор (PK) |
| product_id | INTEGER        | Да           | ID товара, FK → Product.id |
| key_value  | TEXT           | Да           | Строка ключа, уникальная |
| is_sold    | BOOLEAN        | Да           | Продан ли ключ, по умолчанию false |
| order_id   | INTEGER        | Нет          | ID заказа, если ключ продан, FK → Order.id |
| created_at | TIMESTAMP      | Да           | Дата добавления ключа, по умолчанию now() |

## 5. Order (Заказ)

Описание: заказ, оформленный покупателем. Содержит один или несколько товаров (ключей).

### Поля

| Поле        | Тип            | Обязательное | Описание |
|-------------|----------------|--------------|----------|
| id          | SERIAL         | Да           | Уникальный идентификатор (PK) |
| buyer_id    | INTEGER        | Да           | ID покупателя, FK → User.id |

| created_at  | TIMESTAMP      | Да           | Дата создания, по умолчанию now() |

- Почему важно указывать WHERE в UPDATE и DELETE?
Без WHERE команда выполнится для всех строк таблицы. DELETE FROM products удалит все товары, а UPDATE products SET price = 0 обнулит цены всему.

- Что такое первичный ключ (PRIMARY KEY)? Зачем он нужен?
Это столбец (или группа столбцов), который однозначно идентифицирует каждую строку. Он не может быть NULL и должен быть уникальным. Нужен для быстрого поиска записи, связывания таблиц (внешние ключи) и обеспечения целостности данных.

- Пример, когда VARCHAR(20) лучше чем TEXT, и наоборот.
VARCHAR(20) лучше для полей с заранее известной короткой длиной: код валюты (USD), статус заказа (paid). Это экономит место и автоматически защищает от слишком длинных значений. TEXT лучше для длинных или непредсказуемых текстов: описание товара, комментарий, ключ активации. Длина не ограничена, и база данных не резервирует под него лишнее место.
